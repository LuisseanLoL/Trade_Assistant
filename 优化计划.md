# AI Trade Assistant 项目重构与优化全景规划

## 一、 项目架构概览

本项目是一个基于大语言模型（LLM）的量化交易决策支持系统，采用 MVC 架构体系：
* **Model (数据与算法层):** 多源数据爬取（Baostock/Akshare/新浪）+ 核心技术指标计算 + AI 分析引擎（MoA 机制）。
* **View (展现层):** 基于 Dash 框架构建的 Web UI（支持个股与 ETF 双应用面板）。
* **Controller (控制层):** 批处理终端与交互式配置模块。

---

## 二、 当前代码质量评估

### ✅ 核心优势
* **模块化意识良好:** 职责分离清晰，数据获取、AI 交互、UI 渲染初步解耦。
* **创新性架构:** 成功落地 MoA (Mixture of Agents) 多大师议事机制，具备较高的业务价值。
* **多数据源融合:** 整合了传统 API、爬虫（Playwright）与实时行情接口。
* **缓存机制:** 宏观数据、新闻、行情均已实现基础缓存，降低了 API 压力。

### ❌ 亟待解决的痛点 (重构切入点)
* **代码冗余严重 (高):** `app.py` 与 `etf_app.py`、各类 `crawler.py` 之间存在超 60% 的重复代码。
* **解析鲁棒性差 (高):** 严重依赖字符串切片和正则解析 LLM 输出，极易因大模型输出波动而崩溃。
* **同步阻塞与性能瓶颈 (高):** UI 框架与耗时任务（爬虫、LLM 请求）未完全异步隔离；Playwright 每次重新启动浏览器导致极高延迟。
* **异常处理粗糙 (中):** 多处使用裸 `try-except` 捕获异常，缺乏重试机制（特别是针对 LLM 的限流）和精细化日志。
* **类型与工程化缺失 (中):** 缺少静态类型注解（Typing）、配置管理混乱、零自动化测试覆盖。

## 三、 分阶段优化实施路线图

### 第一阶段：代码重构与去重 (第 1-2 周)
**目标：消除冗余，强化底层解析与解耦。**

* **构建公共基类:** * 提取 `base_analyzer.py`、`base_data_crawler.py`、`base_app.py`。
    * 合并个股与 ETF 应用，抽取公共回调逻辑至 `mixins.py`。
* **统一指标计算与策略解耦:** * 将 `calculate_advanced_indicators` 统一提取至 `src/indicators/technical.py`。
    * **严格分离**“指标计算”与“信号生成”，不要在指标计算函数中硬编码交易策略规则。
* **重构 LLM 解析器:** * 弃用正则和字符串截取。强制要求大模型输出严格的 JSON 结构，或使用各大平台最新的 Structured Outputs（结构化输出）特性，确保解析 100% 稳定。

### 第二阶段：类型安全与文档规范 (第 3 周)
**目标：提升代码可读性与 IDE 协作效率。**

* **引入静态类型:** 为核心函数全面添加 `typing` 注解。
* **数据模型化:** 引入 `pydantic` 定义输入输出的数据结构。
* **完善文档:** 使用 Google Style 规范补全 docstring，配置 `mypy` 进行静态类型检查。

### 第三阶段：异常处理、限流与日志体系 (第 4 周)
**目标：提升系统容错率与可观测性。**

* **建立统一异常体系:** 定义 `DataFetchError`、`LLMError` 等自定义异常类。
* **引入结构化日志:** 使用 `structlog` 替代 print，添加请求追踪 ID。
* **LLM 风控与重试机制:** 引入 `tenacity` 库。针对 MoA 机制下的高并发 LLM 请求，配置指数退避算法（Exponential Backoff）以应对 API 的 RPM/TPM 限流。

### 第四阶段：性能与异步化深度改造 (第 5-6 周)
**目标：突破单线程瓶颈，实现真正的非阻塞运行。**

* **剥离后台微服务:** 避免 Dash 的原生机制与 `asyncio` 冲突。引入 Celery 或 RQ 任务队列，将耗时的爬虫和 AI 推理扔到后台，前端仅通过轮询 Redis 获取进度，实现读写分离。
* **爬虫性能飞跃:** * 尽可能通过逆向抓包寻找纯 JSON API 替代 Playwright。
    * 若必须使用 Playwright，必须实现**浏览器上下文池（Browser Context Pool）**，复用实例。
    * 网络请求全面迁移至 `aiohttp`。

### 第五阶段：测试覆盖与持续集成 (第 7 周)
**目标：保障核心逻辑的绝对正确性。**

* **单元测试:** 使用 `pytest` 编写测试用例，确保核心指标计算覆盖率 > 90%。
* **集成与 Mock:** 使用 `pytest-mock` 模拟外部 API 依赖，使用 `vcr.py` 录制 HTTP 请求。
* **代码门禁:** 引入 `black`、`isort`、`flake8` 与 `pre-commit` 钩子。

### 第六阶段：架构升级与高价值数据持久化 (第 8-9 周)
**目标：从单机脚本走向可回测的量化系统。**

* **引入 ORM 持久化:** 使用 SQLAlchemy + PostgreSQL 替代现有的本地 CSV 记录。
* **高价值语料落库:** * 不仅保存买卖决策，必须将 LLM 生成的 `reasoning`（深度思考逻辑）以及触发该决策的**特征截面数据**同步落库。
    * 这为未来进行 AI 归因分析和强化学习（RL）微调积累核心资产。
* **配置管理:** 引入 `pydantic-settings` 统一管理环境变量与 YAML 配置。

### 第七阶段：安全加固与生产监控 (第 10 周)
**目标：满足生产级部署标准。**

* **安全加固:** 废弃字符串拼接路径（改用 `pathlib`），API 密钥实施加密存储。
* **监控告警:** 暴露 Prometheus 指标，集成 Sentry 追踪运行时崩溃。

## 六、 核心风险与应对策略

1.  **大模型并发限流陷阱:** MoA 机制同时呼叫多个 Agent，极易瞬间打满 API 额度。**应对：** 必须在底层封装限流令牌桶或采用 `tenacity` 进行具备抖动（Jitter）的指数避退重试。
2.  **异步与 UI 框架的冲突:** Dash 对原生 Asyncio 支持有限，强行混用会导致事件循环混乱。**应对：** 坚定执行读写分离，耗时计算下放至 Celery 微服务体系。
3.  **爬虫接口失效风险:** 外部金融数据源极不稳定。**应对：** 构建防劣化机制，核心指标获取失败时要有默认的降级策略，不可阻塞整个分析管线。
4.  **重构引入的新 Bug:** **应对：** 遵循“小步快跑”原则，在修改旧代码前，先为该模块写好输入输出的单元测试，确保重构前后结果一致。
